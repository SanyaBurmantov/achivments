## 1. Охват и нагрузка

Период активности: **январь 2025 → июнь 2026** (~16-17 месяцев), без длительных провалов.

Рабочие проекты, вклад по git:

| Проект | Твоих коммитов | Доля в репозитории | Роль | Период |
|---|---:|---|---|---|
| **big-bro** | ~519 | фронтенд целиком | основной / ведущий фронтенд | 2025-01 → 2026-06 |
| **ksm** | 399 | ~10% (большая команда) | фронтенд в команде | 2025-03 → 2026-04 |
| **dnk** | 307 | ~13% | фронтенд в команде | 2025-02 → 2026-05 |
| **law_making** | 91 | малая часть крупного репо | участие | 2025-02 → 2026-02 |

**Итого: ~1370 коммитов по 6 рабочим проектам за ~16 месяцев ≈ ~85 коммитов/месяц.**

### Конкурентность (ключевой показатель нагрузки)
С середины 2025 по начало 2026 одновременно в работе было **4-5 проектов** (big-bro + dnk + law_making + ksm, с августа — expert-journal). Это высокая параллельная нагрузка с постоянным переключением контекста.

**Вывод по объёму:** нагрузка **выше средней**, стабильная, без выгорания в ноль. «Мало» — однозначно нет.

## 2. Технический профиль

Единый, отработанный на всех проектах стек:

- **Frontend:** Vue 3 (Composition API), Vuetify 3, Pinia, Vue Router, Vite, Axios (+ axios-auth-refresh), SCSS/Sass.
- **TypeScript:** применяется в новых проектах (ksm, is_kp) — навык в активном росте.
- **Бэкенд (контекст):** Python/Django на всех проектах — уверенная работа с REST API, JWT-авторизацией, интеграциями.
- **Инструменты/процессы:** ESLint + eslint-plugin-vue, husky (pre-commit), GitLab CI.

**Сильная сторона:** глубокая специализация в одной экосистеме (Vue/Vuetify/Pinia) на 6 проектах подряд — это даёт скорость и предсказуемое качество, а не «по верхам везде».

## 3. Качество работы — что видно по коду (на примере big-bro, флагман)

### Сильные сигналы
- **Архитектурное мышление, а не «фичи».** Повторяющаяся логика вынесена в composables (`useTableList`, `useForm`, `useCompareData`, `useSortByField`, `useTabs`, `useMeta`). Junior так не делает.
- **Переиспользование как система.** Универсальная таблица (`WwTable`) с server-side пагинацией/сортировкой/фильтрацией, синхронизированной с URL; generic CRUD-диалоги; 20+ обёрток над полями ввода с единым UX.
- **Зрелый слой API.** Интерсепторы Axios, централизованная обработка ошибок, прозрачный JWT-refresh с повтором исходного запроса и авто-логаутом.
- **Гигиена процессов.** Линтинг + pre-commit хуки + CI настроены.

### Зоны роста (честно)
- **Тесты отсутствуют** (ни unit, ни e2e). Это главный разрыв до уверенного senior и самый выгодный апгрейд. Composables и `WwTable` — идеальные первые кандидаты на покрытие.
- **Мелкая неаккуратность в проде.** Напр. в `plugins/axios.js` — забытый `console.log`, продублированная строка `axios.defaults.baseURL`. Признак нехватки code review (следствие частой соло-работы).
- **Местами хрупкая логика.** Напр. ветка обработки параметров в `useTableList.getItems` — кандидат на баги. Тесты бы это страховали.

## 4. Объективная оценка специалиста

**Уровень: крепкий middle frontend-разработчик** с senior-замашками в части архитектуры одного проекта.

| Критерий | Оценка |
|---|---|
| Производительность / нагрузка | Выше средней (4-5 проектов параллельно, стабильно 16 мес.) |
| Архитектура и переиспользование | Сильно (composables, generic-компоненты, своя UI-библиотека) |
| Глубина в стеке (Vue/Vuetify/Pinia) | Сильно |
| Самостоятельность / доведение до конца | Сильно (вёл big-bro соло) |
| TypeScript | В росте (есть на новых проектах) |
| Тестирование | Слабо / отсутствует — приоритет №1 для роста |
| Командные процессы / code review | Ограниченный опыт (много соло) |

### План роста до senior (по приоритету)
1. **Тесты** — Vitest + Vue Testing Library (unit на composables/`WwTable`), затем e2e на Playwright.
2. **TypeScript глубже** — довести до уверенного уровня, перенести типизацию на ключевые модули.
3. **Командные практики** — регулярное code review (давать и получать), обсуждение архитектуры.
4. **Производительность под нагрузкой** — виртуализация больших таблиц, code-splitting.

## 5. Как корректно формулировать вклад (чтобы не было натяжки на проверке)

- **«Разработал / вёл как основной фронтенд»** → только **big-bro** (фронтенд целиком — твой).
- **«Разрабатывал фронтенд-часть в команде»** → ksm, dnk, is_kp.
- **«Участвовал»** → law_making, expert-journal.

Это честная формулировка: при проверке git-долей всё сходится, а флагман (big-bro) даёт право говорить о владении архитектурой.
